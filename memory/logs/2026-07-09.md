### Idea 1 — MCPSentry (executable)

**Thesis** (1 sentence): Production agent teams have zero visibility when an MCP tool's schema silently drifts and starts returning wrong answers; ship a hosted contract-testing + canary service that catches drift before it hits prod, sold like Sentry for agent tools.

**ICP** (role + trigger event): AI/platform engineering lead at a Series A–B startup that shipped an agent product to paying customers in the last 90 days, and just experienced a support ticket they traced back to a silently-changed tool response (renamed field, nullable-now-omitted, tightened enum). Pre-trigger they'd never buy this; post-trigger it's an immediate line-item.

**Wedge** (first 12 months): Point MCPSentry at any MCP server URL (or self-hosted). It (1) snapshots the tool schemas, (2) replays a golden fixture set on a cron, (3) diffs live responses against the last-known-good, (4) pages Slack/PagerDuty on any semantic change with a diff, (5) writes a post-drift fixture back so the next canary is stronger. No auto-remediation, no observability platform ambitions, no dashboards to click through — one product: "drift caught before your agent lies to a customer."

**Pain evidence** (2+ permalinks):
  - "The server can change what a tool returns and neither the protocol nor the runtime will raise an exception." — https://hackernoon.com/why-schema-drift-is-the-silent-killer-of-mcp-deployments
  - "A drifted MCP tool gives you a confident wrong answer and an agent that explains why the wrong answer is actually fine." — https://dev.to/nesquikm/my-mcp-tools-broke-silently-schema-drift-is-the-new-dependency-hell-5c49
  - "A tool that returned a nullable field in v0.3 would silently omit it in v0.4." — https://dev.to/sahil_kat/the-mcp-server-ecosystem-in-2026-integration-layer-for-ai-agents-2mln

**Monetization**: $199/mo per monitored MCP server + $99/mo per additional 100 golden fixtures. Target 80% GM (mostly compute for scheduled replays). Unit econ: 30 teams at $299 avg = $9k MRR by month 4 is enough to prove; each team averages ~1.5 servers monitored.

**Distribution** (specific channel + CAC estimate): (a) Open-source a `mcpsentry-oss` CLI that runs the drift-diff locally against a fixture file; get it into the `modelcontextprotocol/servers` GH README "Testing" section via PR — target CAC ~$0. (b) Sponsor 3 Anthropic MCP Discord channels + Latent Space podcast slot — $2k for ~10 signups → CAC ~$200. (c) Show HN launch with a live "Top 100 public MCP servers by drift risk" leaderboard as bait — free.

**Moat** (what compounds): (i) The golden-fixture corpus per customer becomes a switching cost within 30 days of use. (ii) A shared public-MCP drift index (crawler over the 8k+ registry servers) becomes the reference dataset — teams evaluating which MCPs to trust come to your site first. (iii) Trust-graph moat: as your leaderboard becomes canonical, MCP server authors add "MCPSentry Verified" badges = distribution flywheel.

**Why now (2026)**: The Jul-2026 MCP spec release contains breaking wire-level changes (e.g. -32002 → -32602 error codes), the ecosystem crossed 8,000+ registry servers most of which are single-maintainer and can rev whenever, and YC S26's "Software for Agents" RFS is publicly telling every hot batch to build MCPs — meaning your ICP is expanding weekly and every drift incident is Twitter-visible.

**MVP test** (2 weeks): Ship the OSS CLI + a hosted dashboard for 1 MCP server + Slack webhook alerts. Metric: 10 signups from a Show HN post with ≥3 keeping the cron alive after 14 days. If the free tier retains ≥30%, the paid tier will convert.

**Kill criteria** (numeric): <3 paying teams within 60 days of launch → kill. <2 recurring $199+/mo customers after 90 days → kill. If the Show HN thread gets <50 upvotes → distribution isn't real, kill.

**Expansion** (what if it works): (i) Same drift/canary product for OpenAI function-calling schemas, Anthropic tool-use schemas, custom internal agent tools. (ii) Agent-side "evals for tool calls" — hosted eval harness that combines drift + behavior regression. (iii) Sell the drift-index dataset itself to LLM labs training tool-use models. (iv) End-state: the "Datadog for agent tool integrations."

### Idea 2 — Cortex (ambitious)

**Thesis** (1 sentence): On-call SRE runbooks are the highest-consequence tribal knowledge in tech — scattered across Confluence, Slack DMs, PagerDuty notes, and post-mortems — and current runbook tools are still "search + follow steps," so build the agent-native runbook: an on-call copilot that ingests every incident artifact you have and answers "what do I do about this alert right now" in the incident channel.

**ICP** (role + trigger event): SRE Lead or Platform Eng Director at a 50–500-engineer post-Series-B B2B SaaS that just had a P1 within the last 30 days where the on-call engineer couldn't find or execute the correct runbook in time, and the post-mortem action item literally says "consolidate runbooks." Trigger is the post-mortem — the Slack thread where the CTO says "this can never happen again" is your inbound.

**Wedge** (first 12 months): One product, invoked from Slack: `/cortex $incident-id`. It reads the current PagerDuty alert, ingests the customer's Confluence + Slack + past post-mortems + runbook repo at install time, and responds in-thread with (a) the most-relevant runbook, (b) the exact command sequences with tenant-specific placeholders filled from context, (c) related past incidents. No dashboards, no separate app, no incident-management platform.

**Pain evidence** (2+ permalinks):
  - "Manual runbooks rely on tribal knowledge... the engineer who knows the deployment rollback procedure is on vacation." — https://incident.io/blog/runbook-automation-tools-2026-the-complete-guide
  - "Reliance on tribal knowledge means institutional memory fades; documentation endures." — https://rootly.com/incident-response/lifecycle-process
  - YC S26 RFS "Company Brain": "pulls knowledge out of every fragmented source... turns it into an executable skills file for AI." — https://www.ycombinator.com/rfs

**Monetization**: $18/engineer/mo (aggressive vs. incident.io/PagerDuty at ~$30–70) + $5k one-time knowledge-ingest setup fee. Target 75% GM (LLM inference on retrieved context). A 100-eng customer = $21.6k ARR + $5k setup — 3 such customers by month 6 is proof.

**Distribution** (specific channel + CAC estimate): (a) PagerDuty & Slack app marketplace listings — install-friction near zero, discovery via existing on-call tooling searches, CAC ~$50/lead. (b) Sponsor SREcon 2027, on-call-focused newsletters (Gremlin's Failovers, Incident Report Digest) — $5–10k → 3–5 pilots, CAC ~$1.5–2k. (c) Author 5 tightly-targeted "post-mortem teardown" blog posts pointing to a free "runbook audit" (agent scans your runbook repo, returns a readiness report) that becomes the sales wedge — free tool, high intent.

**Moat** (what compounds): (i) Every incident the tool participates in adds a signed post-mortem-derived skill to that customer's private brain — one-year customers have 6–12x more capability than day-one customers = brutal switching cost. (ii) Cross-customer anonymized failure-pattern library becomes proprietary training data no incumbent has. (iii) Deep Slack/PagerDuty integrations create workflow lock-in (every incident channel opens with a Cortex thread; ripping it out is a re-training event).

**Why now (2026)**: (a) LLM context windows crossed the practical threshold to ingest a whole company's runbook estate + relevant Slack in one shot. (b) YC RFS explicitly called out "Company Brain" this cycle, signaling the ICP is now enterprise-buyer-legible. (c) The MCP/Slack agent ecosystem now lets you install and act inside chat with no admin console — the old objection ("we can't add a bot to the incident channel") is dead post-Anthropic-Slack-App standardization.

**MVP test** (2 weeks): Ship for a single design-partner (indie SRE consultant with 5 client accounts). Instrument: (a) time-to-runbook-answered per incident before vs. during pilot, (b) percent of `/cortex` invocations where responder marked "this was right." Target: median TTR-answer drops 60%+, ≥70% "right" rate over 20 invocations. Below either bar → premise is wrong, kill.

**Kill criteria** (numeric): <2 signed pilots within 60 days → kill. <1 paying $18/eng/mo customer at 100+ engineers within 6 months → kill (means the free "runbook audit" wedge doesn't convert). Median TTR-answer improvement <30% at end of MVP → the LLM isn't good enough yet, shelve for 6 months.

**Expansion** (what if it works): (i) Same brain, different rooms — security-incident copilot, customer-escalation copilot, deploy-freeze copilot. (ii) Auto-generate the runbook from the resolved incident (write-back), closing the loop. (iii) End-state: the on-call agent that operates the runbook autonomously with human-approval, i.e. AI-native Site Reliability itself — one of YC's stated multi-billion outcomes.
