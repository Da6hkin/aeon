# vienna-apartments 06:00Z slot silently missing three days in a row

## Symptom

`vienna-apartments` has schedule `0 6 * * *` and `enabled: true` in `aeon.yml`. Per `memory/cron-state.json`:

```json
"vienna-apartments": {
  "last_status": "success",
  "last_success": "2026-07-04T07:36:27Z",
  "last_dispatch": "2026-07-04T07:30:25Z",
  "total_runs": 13,
  "total_successes": 13,
  "success_rate": 1
}
```

Three consecutive daily slots (2026-07-05, 07-06, 07-07 at 06:00 UTC) have failed to update this state. `last_dispatch` is stuck at 2026-07-04 — no dispatch attempt has been recorded, so the skill did not fire (as opposed to firing and failing).

## Why this was filed now

Heartbeat has flagged this pattern across four runs (2026-07-05 08:12Z + 14:29Z + 20:00Z, then 2026-07-06 08:49Z + 15:59Z + 20:13Z) as tracking-only under the 48h dedup rule. Yesterday's 20:13Z heartbeat log explicitly set the escalation rule:

> "If the 07-07 06:00Z slot also fails to fire, file a separate issue — three straight misses would rule out one-off scheduling variance."

The 07-07 06:00Z slot did not fire. Filing this issue moves the signal out of the dedup cycle and onto the public status page.

## Ruling out obvious causes

- **Tick loop is alive**: heartbeat itself ran at 20:13Z on 2026-07-06 and 08:44Z on 2026-07-07, both dispatched via the same tick workflow. `github-trending` also succeeded at 2026-07-07T05:19:21Z. So the Messages & Scheduler workflow is scheduling and dispatching normally.
- **Skill is enabled**: `aeon.yml` line 14 shows `vienna-apartments: { enabled: true, schedule: "0 6 * * *", ... }`.
- **Skill works when it runs**: 13/13 lifetime success rate. When it fires, it succeeds.
- **Not the same class as ISS-001**: `vienna-apartments` is not part of any chain, not a wrapper, and does not share its dispatch path with `overnight-research`.

## Fix

Not yet diagnosed. Next investigative step: reproduce the tick's cron-matching for hour=6, minute=~0-4 at 5-minute tick boundaries. The likeliest bug is around minute-window matching when a tick lands slightly late — e.g. tick at 06:00:23Z reads MINUTE=0 but the catch-up path may already have marked the 06 hour as "already handled" via a state check the current-hour path shouldn't touch.

Prompt-bug angle: worth also confirming that the `-le` numeric comparison at messages.yml:171 handles MINUTE=0 correctly (bash's `-le` should be fine, but has bitten before).

## Verification plan

- Post-fix, next 06:00Z tick should record a fresh `last_dispatch` for `vienna-apartments`.
- Manual `workflow_dispatch` of a tick right now should NOT re-fire vienna-apartments (last_dispatch is 07-04, but hour is 08 not 06 — schedule shouldn't match).
- If the fix lands mid-day, do NOT manually trigger vienna-apartments to force success; let the next 06:00Z tick prove the fix.

## 2026-07-08 update — miss streak broken

Today's 06:00Z slot **fired successfully** without any code change. Per `memory/cron-state.json`:

```
"vienna-apartments": {
  "last_status": "success",
  "last_success": "2026-07-08T06:47:40Z",
  "last_dispatch": "2026-07-08T06:42:01Z",
  "total_runs": 14,
  "total_successes": 14,
  "success_rate": 1
}
```

Both `last_dispatch` (06:42:01Z) and `last_success` (06:47:40Z) advanced. Skill produced 11 qualifying listings and delivered its Telegram notification via the standard post-run pending-notify path (see 2026-07-08 log entry).

**Interpretation.** The 3-day miss streak (07-05, 07-06, 07-07) was intermittent, not a consistent bug. Nothing in `messages.yml`, `chain-runner.yml`, or `aeon.yml` changed between yesterday and today, so whichever candidate hypothesis was correct, it was not deterministic. Two most consistent explanations left standing:

- **H1 (tick-drift near hour boundary):** Some 06:00Z ticks land 4+ minutes late, past the catch-up window at `messages.yml:171` (`[ "$C_MIN" -le "$MINUTE" ]`). Today's tick may have landed earlier / more punctually than the last three days'. Consistent with the observed 06:42Z dispatch — which is 42 minutes into the hour, comfortably inside the window from a hypothetical tick that arrived at 06:40Z.
- **H4 (silent workflow_dispatch API failure):** GitHub Actions dispatch API had transient errors on 07-05/06/07 and recovered today. Untestable retroactively.

Hypotheses H2 (dispatch cap) and H3 (var parsing) are effectively ruled out — nothing about the aeon.yml entry, the enabled skill list, or the var string changed between 07-07 (missed) and 07-08 (fired). If either were the cause, today would have missed as well.

**Status.** Not resolving the issue outright — a one-day recovery doesn't rule out the underlying pattern. Downgrading severity from `medium` to `low` and switching to watch mode:
- Watch 2026-07-09, 07-10, 07-11 06:00Z slots.
- If all three fire cleanly → close as `wontfix` (intermittent, non-reproducible; no user-visible impact after 3-day miss).
- If 1+ slot misses → escalate back to `medium` and instrument `messages.yml` cron-matching for logging around hour boundaries.
