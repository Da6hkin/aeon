# chain:overnight-research keeps failing; startup-idea downstream state never updates

## Symptom

`chain:overnight-research` (schedule `45 4 * * *`, `on_error: continue`) has now failed on two consecutive days:

- **2026-05-21T05:57:55Z** — first observed failure
- **2026-05-22T05:45:59Z** — re-occurred on today's 04:45Z fire (only ~36s after the chain's wrapper `overnight-research` reported success at 05:45:23Z)

The wrapper `overnight-research` entry succeeds on each run, but the chain orchestrator's tracked status flips to `failed` within a minute, suggesting a downstream step in the chain pipeline errors out (or returns non-zero) and bubbles up to the chain-level state even with `on_error: continue`.

A downstream signal corroborates this: `startup-idea` (schedule `0 */6 * * *`, runs as the second step in the chain) has `last_success: 2026-05-15T01:39:05Z` — **7 days stale**. Every chain run since 2026-05-15 has failed to update its state. So either:

1. `startup-idea` itself is what's erroring (the most likely failing step), OR
2. The chain runner is dispatching `startup-idea` but the per-skill state-update step never fires when the chain reports failure

Either way, `startup-idea` has effectively been silently broken since 2026-05-15.

## Why this was filed now

Heartbeat has flagged this pattern in **5 consecutive runs** (2026-05-20 20:00Z + all three 2026-05-21 slots + today 08:00Z) and suppressed every notification under the 48h dedup rule. Filing this issue moves the symptom out of the dedup cycle and into a tracked artifact that:

- Surfaces on the public status page (`docs/status.md → Open issues`)
- Will be picked up by `skill-repair` (reactive, triggers on chronic failure)
- Stops the daily "same symptom, suppress" pattern from masking a real regression

## Root cause diagnosis (2026-07-05)

Diagnosed during the daily wrapper `skill: overnight-research` invocation. Two independent workflow bugs — see `root_cause` in frontmatter for the full write-up.

**Bug A — chain-runner.yml jq regex.** Line 63 filter `test("skill: ${skill}("; "i")` contains an unmatched `(`, so run_id discovery always returns empty. `set -e` aborts at Step 1/4 after the 60s poll. Reproduction:
```
$ jq 'test("skill: github-trending("; "i")'
jq: error: Regex failure: end pattern with unmatched parenthesis
```
Timing match: chain run #28730288934 died at exit 1 exactly 67s after start.

**Bug B — messages.yml skill parser.** The parser regex `^\ \ ([a-zA-Z0-9_-]+):$` matches both skill entries under the top-level and chain entries under `chains:` (same indent depth). Every 04:45Z fire double-dispatches:
- `chain: overnight-research` via chain-runner.yml (proper)
- `skill: overnight-research` via aeon.yml (bogus — no `SKILL.md`)

## Fix

Both patches are trivial:
- **chain-runner.yml L63:** replace regex with exact/startswith equality — `select(.displayTitle == "skill: ${skill}" or (.displayTitle | startswith("skill: ${skill} (")))`.
- **messages.yml (~L109):** track `IN_CHAINS_SECTION` boolean, skip skill matching while inside `chains:` block.

## Verification plan

- Post-merge, next 04:45Z fire should produce exactly ONE run (`chain: overnight-research`), not two.
- Chain should reach Step 4/4 or fail on a genuine downstream step (not the polling bug).
- `startup-idea` state should update within one cycle.

## 2026-07-09 — RESOLVED: both bugs fixed at the source

Fixed properly from a Claude Code session that has `workflows` scope (the blocker
that stalled this issue for weeks — the Actions default token could not push
workflow files — does not apply here). Both patches are minimal and were verified
by extracting the affected shell logic and running it against the real `aeon.yml`.

**Bug A — `.github/workflows/chain-runner.yml` (dispatch_skill run-id lookup).**
Replaced the broken jq filter

```
select(.displayTitle | test("skill: ${skill}("; "i"))     # unmatched ( → jq errors → empty
```

with an exact / prefix match against the real run-name format (`skill: <name>` or
`skill: <name> (<var>)`):

```
select(.displayTitle == "skill: ${skill}" or (.displayTitle | startswith("skill: ${skill} (")))
```

Also raised `gh run list -L 10` → `-L 20` so a run isn't missed when a parallel
step dispatches several skills at once. The chain can now discover its dispatched
run IDs, so it reaches Step 4/4 instead of aborting at Step 1 under `set -e`.

**Bug B — `.github/workflows/messages.yml` (skill parser).**
Added an `IN_SKILLS` section guard so the parser only reads entries under the
top-level `skills:` block. Previously it scanned the whole file, so any
2-space-indented key elsewhere — the `overnight-research:` chain, `jsonrender:`
under `channels:`, `provider:` under `gateway:` — registered as a phantom skill.
`overnight-research` even inherited the chain's `schedule: "45 4 * * *"` and fired
a daily bogus `skill: overnight-research` run. Verified against `aeon.yml`:
`overnight-research`, `jsonrender`, and `provider` no longer parse as skills; the
real skill set is unchanged.

**Verification.** Local simulation confirms the chain-runner filter matches
`skill: startup-idea` (with and without a var suffix) and the skill parser drops
the three phantoms. Live confirmation on the first post-merge 04:45Z fire:
exactly one `chain: overnight-research` run (no ghost `skill:` run), the chain
reaches Step 4/4, and `startup-idea` / `idea-capture` / `morning-brief` state
advances within one cycle.

**Note (separate, not fixed here):** `startup-idea`'s standalone `0 */6 * * *`
schedule never actually cron-matches because its multi-line brace entry in
`aeon.yml` puts `schedule:` at 6-space indent while the parser only reads
`schedule:` at 4-space indent — so `SKILL_SCHEDULE[startup-idea]` is empty and the
`[ -z "$SCHED" ] && continue` guard skips it. In practice startup-idea has only
ever run via the chain, so restoring the chain (Bug A) restores it. Left as-is to
avoid changing dispatch cadence the operator didn't ask about; flagged for
follow-up if standalone 6-hourly runs are desired.
